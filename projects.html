Plant Swap

Keywords: Java, Maven, SQL, Spring Boot, PostgreSQL, Postman, Docker, Back-End, Solo Project

This project is a Plant Swap service where private individuals can swap or buy plants from one another.


The application is coded in Java and is a Spring boot REST API with Maven using a PostgreSQL database and run via a Docker container. The application is backend-only and is best experienced using the Postman documentation linked in the article.

The application has three main model classes: (1) User, (2) Plant and (3) Transaction which all have CRUD implementation and for which data are stored in three separate PostgreSQL database tables. Each model class has a related service, repository and controller class. The database has a normalization of 3N , with very low data redundancy and the application has very robust validation and error handling

GitHub Repository

The Plant Swap service allows for registration, update, deletion of users (User class), plant listings (Plant class) and purchase/swap requests (Transaction class). A plant listing must either be for swap (and then have swap condition listed) or for purchase (and then have a price). Transactions for plants that are for purchase are accepted automatically, while transactions that are swap offers must be accepted or rejected manually via API end-points.

The project has a very robust validation implementation which checks that all user input (RequestBodies and RequestParameters) is valid and otherwise cast appropriate error messages. Use of global error handling ensures that errors caught and handled in uniform manner.

The User entity has a one-to-many relationship to both the Plant entity (plant listing) and the Transaction entity. Furthermore, the Plant has a one-to-many relationship with Transaction, since a transaction request can be rejected, and it should then be possible for another transaction to be made for that same plant listing.

Postman Documentation

In general, the project has delete on cascade for the database. However, there are many method-checks for if a delete request is allowed. For example, if a plant listing has a pending transaction request, deletion of that plant is not allowed and attempts to delete it will cast an error message informing the user that pending transactions must be resolved before it can be deleted.

AirBnB Clone

Keywords: Java, Maven, NoSQL, Spring Boot, Spring Security, JWT tokens, MongoDB Atlas, Postman, Docker, Back-End, Group Project, Agile Workflow

This project is a Living Space Rental service (named AirBnB Clone - the magic is in the code, not the name!) where private individuals can rent-out and rent accommodations from one another.


The application is coded in Java and is a Spring boot REST API with Spring Security implementation and Maven. It uses MongoDB as a database and run via a Docker container. The application is backend-only and is best experienced using the Postman documentation linked in the article. This project is a group project coded with two other talented individuals.

This project is a Living Space Rental service (named AirBnB Clone - the magic is in the code, not the name!) where private individuals can rent-out and rent accommodations from one another. The application is coded in Java and is a Spring boot REST API with Spring Security implementation and Maven. It uses MongoDB as a database and run via a Docker container. The application is backend-only and is best experienced using the Postman documentation linked in the article. This project is a group project coded with two other talented individuals.

GitHub Repository

This application is modelled after AirBnB, and allows users to publish and manage listings, make bookings and leave reviews. The application has four main model classes (User, Listing, Booking, Review) and uses DTOs to package and deliver information between the controller classes and the service classes (where the DTOs are mapped to and from the different model classes). A user can own several listings, bookings and reviews, and therefor has a one-to-many relationship to all of these. Similarly, a listing can have many bookings and reviews and therefor has a one-to-many relationship to these.

Since the database used is a NoSQL database, the data is not normalized and things such as username and listing title are sometimes double stored in bookings or reviews, to allow for e.g., showing the username of the person leaving the review without retrieving data from any source other than the review entity itself. During development we used a MongoDB Atlas cluster in order for the whole team to work with the same database.

Postman Documentation

The application has a comprehensive Spring Security implementation, which uses e.g., filter chains, JWT tokens, CORS configuration and password authentication. The application has end-point access regulation based on if the current user is logged in and (if they are) what roles they have. Non-logged in user has the lowest access level , and can only browse the listing in the database (along with associated reviews and host profile), as well as register a new user or log in. A logged in user with the role admin has the. highest access level and can manage their own profile, listings, bookings and reviews, as well as delete those of other users.

The project has a very robust validation implementation which checks that all user input (RequestBodies and RequestParameters) is valid. This is done both through the @Valid annotation and through checks in the service class methods. Use of global error handling ensures that errors caught and handled in uniform manner.

Personal Website

This website is a front-end only application coded in html, CSS and pure JavaScript. It has responsive design that adapts the layout according to the view port screen, using media queries for three different minimum view port widths. It also includes a contact form with validation and numerous cool hover and button click effects.

GitHub Repository

Why not try it for yourself?

Narrow down the window width and see the design change at 900px, 575px and 400px break points. The break points are set according where they need to be in order for the layout to always remain pleasing and functional.

At 575px the secondary navigation menu (circular icon buttons) moves to the top and the main menu (and secondary menu) becomes fixed at its top positions even when scrolling. At 400px the main menu splits the four options into two rows.

The secondary menu consists of circular buttons with white icons, when hovering over a button, it expand to show a the title of the menu option. When the button is clicked the button is fixated in its exanded layout form using JavaScript functions. The hover effects are done using both CSS and JavaScript in combination.

The contact form on the Contact page is sadly not functional as a contact form (i.e., the message is not sent anywhere - as of yet!), but it does have form validation through JavaScript, which checks that the user has filled out all fields before clicking submit. Try skipping a field and see what error message you get!
